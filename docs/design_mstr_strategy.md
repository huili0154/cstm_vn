## 1. 策略概述

MSTR 是一个面向 A 股 ETF 标的池的 **Tick 级均值回归轮动策略**。

核心思想：在一组高相关性 ETF 中，利用品种间价格比率（Ratio）的历史均值回归特性，在 Ratio 偏离均值超过阈值时进行换仓操作 —— 卖出相对"贵"的，买入相对"便宜"的。

**核心特性**：

| 特性       | 说明                                   |
| -------- | ------------------------------------ |
| **信号计算** | N-1 天历史日线 + 当日 Tick 价格融合为 N 点窗口，统一计算 |
| **执行时点** | 盘中实时，信号触发即执行                         |
| **持仓模式** | 多标的分仓，按信号强度分配权重                      |
| **下单方式** | Block 分块交易，冷却期控制节奏，双边激进挂单            |
| **风控**   | 单标的仓位上限 + 冷却期 + 信号冲突检查 + T+1          |

### 1.1 关键设计理念

- **全员参与**：标的池内所有品种始终参与信号计算，不做预筛选，无 watchlist。每个 tick 到来时对所有品种对实时评估。
- **日线与 Tick 统一**：当前 tick 价格视为滚动窗口的"第 N 天"，与前 N-1 天日线数据合并，直接计算偏离度和得分。信号模型只有一套数学公式，不区分"日线阶段"和"Tick 阶段"。
- **Block 分块**：总资产等分为若干 Block，每次交易以一个 Block 为单位。冷却期控制节奏，T+1 是天然风控。简单直接，无复杂分批状态机。

## 2. 信号计算模型

### 2.1 价格比率（Ratio）

对于标的池中任意两只标的 A 和 B，定义价格比率序列：

```
Ratio_t = AdjPrice_A(t) / AdjPrice_B(t)
```

其中：

- **历史部分**（t = 1 ... N-1）：使用后复权日线收盘价（close\_bwd）
- **当日**（t = N）：使用 tick 实时价格 × adj\_factor（见 2.5 节）

### 2.2 滚动统计量

基于前 N-1 天的 Ratio 历史序列（不含当日）预计算：

```
μ      = mean(Ratio_1 ... Ratio_{N-1})     # 均值
Max    = max(Ratio_1 ... Ratio_{N-1})       # 最大值
MaxDev = Max - μ                            # 最大偏离幅度
```

μ 和 Max **每日开盘前计算一次，盘中不变**。

### 2.3 实时偏离与得分

盘中每个 tick 到来时，用当日实时价格更新 Ratio\_N，然后计算：

```
Dev   = Ratio_N - μ       # 当日偏离
Score = Dev / MaxDev       # 归一化得分（0~1+ 范围）
```

Score 随 tick 价格实时变动。Score > 1.0 意味着当前偏离已超过历史窗口内的最大偏离。

### 2.4 信号触发条件

对于持仓标的 H 和标的池中每个其他标的 Si，计算 Score(H, Si)。当以下条件**全部满足**时产生换仓信号：

1. `Dev(H, Si) > 0` — H 相对 Si 变贵了（Ratio 上升）
2. `MaxDev(H, Si) > 0` — 历史存在波动空间（分母合法）
3. `Score(H, Si) > K_threshold` — 归一化偏离超过阈值
4. `Dev(H, Si) > LeastBias` — 绝对偏离超过噪音门槛，避免价差极小时的误触发

满足条件 → 应该减持 H，增持 Si。

### 2.5 价格一致性处理（adj\_factor）

历史 Ratio 用后复权价（close\_bwd = raw\_close × adj\_factor）计算。盘中 Ratio 用 tick 实时价格。为了可比，需要将 tick 价格转换到后复权口径：

```
Ratio_N = (last_price_A × adj_factor_A) / (last_price_B × adj_factor_B)
```

adj\_factor 在 on\_day\_begin 时从当日 BarData 获取，日内不变。这样 Ratio\_N 与历史 Ratio 在同一口径下计算，μ、Max 直接可用。

### 2.6 Score 与 Dev 的角色分工

Score 和 Dev（Bias）在策略中扮演不同角色：

- **Score = Dev / MaxDev** — **触发指标**：决定"是否应该换仓"。它衡量当前偏离在历史波动范围内的相对位置。Score > K 意味着偏离进入历史高位区间，换仓的统计概率有利。
- **Dev = Ratio\_N - μ** — **利润指标**：决定"这次换仓能赚多少"。Dev 是实际的价差幅度，如果 Ratio 回归均值 μ，Dev 就是本次换仓可捕获的理论收益（扣除佣金前）。

两者的配合关系：

| Score  | Dev             | 含义                              |
| ------ | --------------- | ------------------------------- |
| 高（> K） | 大（>> LeastBias） | **最佳机会**：统计确认可靠，利润空间大           |
| 高（> K） | 小（≈ LeastBias）  | 统计上该操作，但利润被佣金吃掉 → LeastBias 过滤掉 |
| 低（< K） | 大               | Dev 大但在历史波动范围内算正常 → 不操作         |
| 低（< K） | 小               | 无信号                             |

**设计原则**：

- Score 是操作的触发器（门槛），决定是否行动
- Dev 是利润的度量尺，决定行动的力度和优先级
- LeastBias 是保护阈值，过滤掉 Score 达标但 Dev 太小不值得操作的情况
- 当多个候选同时触发时，**按 Dev 降序**分配资金（而非 Score），因为 Dev 直接反映利润潜力

### 2.7 卖出与买入的决策分离

卖出和买入是两个**独立的决策过程**，不应耦合。

**卖出决策**：对每个持仓标的 H

- 遍历标的池中**所有非 H 的标的** Si（无论 Si 是否已有持仓）
- 计算 Score(H, Si)，只要存在任何一个 Si 使得四条触发条件全部满足 → H 应该被卖出（部分或全部）
- 卖出量由执行阶段决定，卖出决策不关心资金去向

**买入决策**：卖出释放资金后（或通过信用额度获得买入能力时，见 3.2.3 节）

- 遍历标的池中**所有非 H 的标的** Si（包括已部分持仓的）
- 每个 Si 的"吸引力" = 以所有持仓标的为 H 计算的 Score(H, Si) 中的最大值
- 按 **Dev 降序**排列（Dev 大 = 利润空间大），在 Dev 相近时参考 Score
- 已持仓标的也可以作为买入目标 — 上一次只买了一部分，现在 Score 仍然达标，应该加仓

**为什么不能只看"持仓 vs 非持仓"对**：

- 场景 A：持仓标的 X 和 Y，X 相对 Y 变贵了 → 应该减 X 加 Y。如果只看非持仓标的，会错过这个机会。
- 场景 B：上一轮买入 Z 只完成了第 1 批（30%），Z 的 Score 仍在走高 → 应该继续加仓 Z。如果 Z 不在"非持仓"集合中，加仓逻辑失效。

> **计算范围**：每个 tick 对所有持仓标的 H，计算 H 对池中每个非 H 标的的 Score。对于持仓 3 只、池 21 只，每 tick 计算 3×20 = 60 个 Score。全量两两是 21×20 = 420。60 个浮点运算在微秒级完成，无性能问题。

### 2.8 信号冲突安全检查（Netting Check）

由于 Block 串行执行（同一时刻只有一个活跃 Block，见 3.5.8 节），每次只选一个最优的 (H → Si) 对执行，理论上不会出现同一标的同时被买入和卖出的矛盾。但作为防御性安全检查，策略在信号收集阶段仍然检测以下异常模式：

**检测的冲突模式**：

- **互斥冲突**：持仓 A、B。信号"卖 A 买 B"和"卖 B 买 A"同时满足触发条件。这说明 A/B 之间的偏离方向在不同参照系下产生了矛盾判断。
- **链式冲突**：信号"卖 A 买 B"和"卖 B 买 D"同时触发。B 既是买入目标又是卖出候选。

**处理方式**：

- **不影响执行**：择股逻辑（3.4 节）始终选 Dev 最大的单一 (H, Si) 对执行，冲突不改变执行结果
- **记录日志**：检测到冲突时记录一条 NETTING_CONFLICT 日志，包含所有冲突信号对及其 Score/Dev
- **事后分析**：回测结束后统计冲突出现频率和当时的市场条件，判断信号模型是否存在系统性问题

> **设计说明**：Netting Check 纯粹是一个安全监控机制，与执行流程无关。如果回测中从未出现冲突日志，说明信号模型的一致性良好。如果频繁出现，需要检查触发条件或窗口参数。

## 3. 执行层（Execution Layer）—— Block 资金分块模型

### 3.1 设计思路

放弃原来的"三阶段分批建仓"方案（Phase 1/2/3 + W1/W2/W3），改为更简洁的 **Block 资金分块模型**：

- 将系统总资产等分为若干个 Block（资金块），每次交易的目标就是成交一个 Block
- 信号触发即下单一个 Block，逻辑简单直观
- 通过冷却期（Cooldown）控制同一天内多次下单的节奏
- 订单执行采用"双边激进挂单 → 一边成交即撤另一边 → 改价争取更优成交"的策略

### 3.2 Block 的定义与管理

#### 3.2.1 Block 大小

Block 的大小由**当前现金仓位中剩余的 Block 数量**决定，而非总资产：

```
cash_blocks = num_blocks - Σ block_count[symbol]   # 现金中剩余的 Block 数
block_size  = (当前现金 - min_cash_reserve) / cash_blocks  # 每个 Block 的实际金额
```

| 项目                   | 说明                             |
| -------------------- | ------------------------------ |
| num\_blocks          | 总分块数（如 5），策略参数，策略生命周期内不变       |
| cash\_blocks         | 当前现金中的 Block 数 = num\_blocks - Σ block\_count[symbol] |
| block\_size          | 动态值，= (当前现金 - min\_cash\_reserve) / cash\_blocks |
| block\_count[symbol] | 每只标的上堆积的 Block 数量               |

**示例**：总资产 50 万，num\_blocks = 5，初始全部在现金。

| 操作         | cash\_blocks | 现金    | block\_size | 说明                       |
| ---------- | ----------- | ----- | ---------- | ------------------------ |
| 初始状态       | 5           | 50 万  | 10 万       | 50 万 / 5                 |
| 买入 A 一个 Block | 4           | 40 万  | 10 万       | 40 万 / 4                 |
| 买入 B 一个 Block | 3           | 30 万  | 10 万       | 30 万 / 3                 |
| A 下跌，现金不变  | 3           | 30 万  | 10 万       | block\_size 不受持仓涨跌影响      |
| 追加买入 A     | 2           | 20 万  | 10 万       | 20 万 / 2                 |

> **关键设计**：Block 的核心是**计数**而非金额。任何时刻只需知道：现金占几个 Block、每只股票占几个 Block，总和恒等于 num_blocks。至于某只标的当前市值是涨了还是跌了，不影响 Block 计数和分配逻辑。block_size = (现金 - min_cash_reserve) / cash_blocks 意味着每次买入消耗的是可用现金中等分的一份，不受持仓涨跌干扰。
>
> **关于串行与现金冻结**：串行原则（3.5.8 节）保证同一时刻只有一个活跃 Block 交易。Block 启动时计算好 block_size 后，在该 Block 完成前不会启动新的 Block，因此不存在"冻结现金导致 block_size 计算偏差"的问题。未来如果开启多 Block 并行交易，需要将冻结中的现金从可用现金中扣除后再计算 block_size。
>
> **cash\_blocks = 0 时的处理**：当所有 Block 都在持仓中（cash\_blocks = 0），block\_size 公式会除零。此时策略行为：
> - **换仓模式仍然可以执行**：换仓不需要额外现金 — 卖出 1 Block 的 H 同时买入 1 Block 的 Si。买入量由卖出侧决定：`buy_volume = round_down(sell_expected_amount / buy_price, 100)`，其中 `sell_expected_amount = 持仓总量 / block_count[H] × sell_price`。cash\_blocks 在换仓中不变（卖 -1 的同时买 +1）。换仓过程中买单先于卖单成交时，现金可能暂时为负数，由信用额度机制保障（见 3.2.3 节）。
> - **纯买入不可执行**：没有剩余现金 Block，不启动纯买入。信号记录但不下单。

#### 3.2.3 信用额度（Credit Line）

换仓 Block 的买卖两侧同时提交订单。由于卖单尚未成交，账上可能暂时没有足够现金支付买单。为避免买单被引擎拒绝，系统在引擎层引入了**信用额度**机制：

```
credit_line = 总净值 × credit_ratio
总净值     = 现金余额 + 冻结资金 + Σ(持仓量 × 市价)
```

| 项目 | 说明 |
| --- | --- |
| credit\_ratio | 信用额度比例（0~1），引擎参数，默认 0.5 |
| credit\_line | 当前可用信用额度（元），随净值实时变化 |

**核心规则**：引擎在判断买单资金是否充足时，根据 credit\_ratio 是否大于 0 分为两种模式：

- **未开启信用（credit\_ratio = 0）**：可用资金 = 账户可用现金。现金不足即拒单，现金不会为负数。这是传统模式。
- **开启信用（credit\_ratio > 0）**：可用资金 = 账户可用现金 + credit\_line。只要买单金额在信用额度之内，即使现金不足也允许下单。买入消耗现金后，如果卖单尚未成交，现金暂时为负 — 只要负值的绝对值不超过 credit\_line，下单不被拒绝。
- **信用基于总资产**：信用额度基于总净值（现金 + 冻结 + 股票市值），不依赖某个特定的待成交卖单。只要账户有资产 — 无论是现金还是股票 — 都可以获得信用额度。
- **自动回收**：卖单成交后现金回正，信用额度自然释放。正常情况下，换仓 Block 完成后现金余额接近换仓前水平。

**典型场景**：总资产 30 万全部在股票中（现金 ≈ 0），credit\_ratio = 0.5，触发换仓信号（卖 A 买 B）：

| 时点 | 现金 | credit\_line | 可下单？ |
| --- | --- | --- | --- |
| 换仓前 | ≈ 0 | 15 万（30万 × 0.5） | — |
| 买单提交，成交 | −10 万 | ≈ 15 万 | ✓（\|−10万\| < 15万）|
| 卖单随后成交 | +0.2 万 | ≈ 15 万 | — |

> **与 Block 模型的关系**：信用额度是引擎层面的资金保障机制，不改变 Block 计数和分配逻辑。cash\_blocks = 0 时纯买入仍然不启动（因为没有可分配的 Block），但换仓模式下的买侧订单不再因现金不足而被拒绝。

#### 3.2.2 卖出时的 Block 计算

卖出以 Block 为单位。已知标的上有 block\_count 个 Block，卖出 1 个 Block 的量为：

```
sell_amount = 持仓总量 / block_count
sell_volume = round_down(sell_amount, 100)   # 对 100 股取整（A 股最小交易单位）
```

| block\_count | 卖 1 Block | 说明      |
| ----------- | --------- | ------- |
| 1           | 全部卖出      | 清仓该标的   |
| 2           | 卖出持仓的 1/2 | 保留一半    |
| 3           | 卖出持仓的 1/3 | 保留 2/3  |

卖出后：block\_count[symbol] -= 1，cash\_blocks += 1，回收的现金纳入现金池。

> **注意**：对 100 股取整会导致实际卖出量与理论值有偏差，尤其在持仓量较小时。这是 A 股交易的固有约束，不需要特殊处理。

### 3.3 冷却期（Cooldown）

同一个交易日内，设定两级冷却期控制下单节奏：

| 阶段       | 触发条件                     | 冷却时长            | 说明                       |
| -------- | ------------------------ | --------------- | ------------------------ |
| 首次冷却     | 当日第 1 个 Block **完成**后     | cooldown\_1 分钟  | 确认首笔交易后的市场反应             |
| 后续冷却     | 当日第 2 个及之后每个 Block **完成**后 | cooldown\_2 分钟  | 持续控制下单节奏                 |

**计时起点**：冷却期从 Block 交易**完全结束**（状态变为 DONE、TIMEOUT 或 CRITICAL）的时刻开始计时，而非从第一个成交回报开始。Block 从被动挂单到最终完成通常在几秒~几十秒，极端情况下可能达到 block\_timeout，冷却期在此之后才启动。

**时钟来源**：所有计时（冷却期、block\_timeout、chase\_wait\_ticks）统一使用 `tick.datetime` 作为时钟源。这确保回测和实盘行为完全一致 — 回测中用历史 tick 时间戳推进时钟，实盘中 tick 时间戳由交易所提供。不使用系统时钟 `datetime.now()`。

**流程**：

```
第 1 个 Block 交易（被动→激进→追价→DONE）
    → 首次冷却开始计时（等 cooldown_1 分钟）
        → 冷却结束，第 2 个 Block 交易
            → Block 完成 → 后续冷却开始计时（等 cooldown_2 分钟）
                → 冷却结束，第 3 个 Block 交易
                    → Block 完成 → 后续冷却开始计时（等 cooldown_2 分钟）
                        → … 以此类推，每笔完成后都等待 cooldown_2
```

**设计意图**：

- 首次冷却（cooldown\_1）是"试探后观望"：第一笔交易可能基于尚不稳定的信号，等待一段时间确认市场方向
- 后续冷却（cooldown\_2）是"持续节奏控制"：每笔交易完成后都需等待，防止连续密集下单

> **冷却期计数是全局的**：不区分买卖方向或标的，当日所有 Block 交易累计计数。

### 3.4 择股原则

信号触发时，可能有多个候选标的同时满足条件。选择规则：

1. 所有满足触发条件的候选按 Dev 降序排列
2. **最佳候选**：Dev 最大的标的
3. **近优候选**：Dev 在最佳候选的 (1 - δ) 范围内的标的也有资格参与
   - 例如 δ = 0.1，最佳 Dev = 0.05，则 Dev > 0.045 的候选都入围
   - 这允许分散到多个相近标的，降低集中度风险
4. 候选数量受 max\_positions 限制

**近优候选的实际用途**：串行 Block 模型下每次只执行一个 (H, Si) 对。当最佳候选的标的已达到 max\_block\_per\_symbol 上限时，自动降级选择下一个近优候选。近优候选也用于分散建仓 — 如果连续多个 Block 都选同一只标的，近优候选机制允许后续 Block 分散到 Dev 相近的其他品种。

### 3.5 单 Block 订单执行逻辑

当信号触发且不在冷却期内时，执行一个 Block 的交易。交易分为两种模式：

- **换仓模式**（卖 H 买 Si）：双边操作，使用子单拆分 + Fill Matching 策略（3.5.1 ~ 3.5.6）
- **纯买入模式**（仅买 Si）：单边操作，空仓建仓或有剩余现金 Block 时触发（3.5.9）

以下 3.5.1 ~ 3.5.8 描述换仓模式的完整流程。

#### 3.5.1 子单拆分（Sub-lot Splitting）

一个 Block 的总量拆分为 **sub\_lots** 个子单（默认 5），每个子单独立提交、独立撤销。

```
total_buy_volume  = round_down(block_size / buy_price, 100)
total_sell_volume = round_down(持仓总量 / block_count, 100)

# 拆分为 sub_lots 个子单
sub_buy_volume  = round_down(total_buy_volume / sub_lots, 100)
sub_sell_volume = round_down(total_sell_volume / sub_lots, 100)

# 最后一个子单吸收取整余量
last_buy  = total_buy_volume  - sub_buy_volume  × (sub_lots - 1)
last_sell = total_sell_volume - sub_sell_volume × (sub_lots - 1)
```

**为什么拆分为子单**：

- 子单是撤单的最小操作单位 — 不需要对一个大单做"部分撤单"（交易所不支持部分撤单）
- 当需要撤销某个金额时，撤掉若干个完整子单即可，操作干净
- 子单越多粒度越细，但太多（>10）会碎片化且可能触碰交易所频率限制；太少（<3）粒度太粗

> **sub\_lots = 5 的理由**：提供 20% 粒度。当需要撤销另一边的"大约 40% 未成交量"时，撤 2 个子单即可。相比子单 = 3（33% 粒度），5 更灵活。

#### 3.5.2 初始下单

双边各提交 sub\_lots 个子单，均使用**被动价格**（挂单等待），争取更优成交价：

| 方向 | 限价                    | 含义                  |
| -- | --------------------- | ------------------- |
| 买入 | bid\_price\_1（买一） | 以买方最优价挂单，被动等待成交 |
| 卖出 | ask\_price\_1（卖一） | 以卖方最优价挂单，被动等待成交 |

双边各 sub\_lots 个子单同时提交，均以相同的被动价格挂单。

#### 3.5.3 累计成交量匹配（Fill Matching）

放弃"子单 #i 配对子单 #i"的配对模型。子单只是订单管理的单位（方便撤单），**匹配逻辑基于买卖两边的累计成交量差额**。

**核心状态**：

```
BlockTrade:
    buy_target:     int     # 买入目标总量
    sell_target:    int     # 卖出目标总量
    buy_filled:     int     # 买方累计已成交量（含被动 + 激进）
    sell_filled:    int     # 卖方累计已成交量（含被动 + 激进）
    buy_aggressive: int     # 已提交的激进买入量
    sell_aggressive:int     # 已提交的激进卖出量
```

**触发规则**：每次收到成交回报（on\_trade）时，更新 buy\_filled 或 sell\_filled，然后计算差额：

```
buy_excess  = buy_filled  - sell_aggressive   # 买方已成交但卖方尚未激进补单的量
sell_excess = sell_filled  - buy_aggressive   # 卖方已成交但买方尚未激进补单的量
```

- `buy_excess > 0`：买方多成交了 buy\_excess，需要对卖方提交 buy\_excess 的激进卖单
- `sell_excess > 0`：卖方多成交了 sell\_excess，需要对买方提交 sell\_excess 的激进买单

**执行动作**：当 excess > 0 时（以 buy_excess > 0 为例，需要激进卖出）：

1. **确定撤单目标**：需要从卖方被动子单中撤出至少 excess 量的未成交股数
2. **撤单优先级 — 先撤完全未成交的子单，后撤部分成交的子单**：
   - 优先撤**完全未成交**（remaining = volume）的子单：这些子单没有任何成交记录，撤掉不浪费排队优势
   - 如果完全未成交的子单撤完仍不够，再撤**部分成交**的子单：这些子单已有一部分被动成交（拿到了好价格），但剩余量必须让出来给激进单
   - **已全部成交**的子单不撤（remaining = 0）
3. **计算实际撤出量**：累计所有被撤子单的 remaining 之和
4. **提交激进补单**：以 bid_price_1（激进卖出价）提交一个新订单，量 = excess
5. **处理多撤量**：如果实际撤出量 > excess，多出的部分以 ask_price_1（被动价）重新挂一个新子单
6. **更新计数**：sell_aggressive += excess

**极限场景示例**：sub_lots = 3，每个子单 = 2000 股

- 买方：#1 全部成交 2000，#2 全部成交 2000，#3 成交 1000/2000 → buy_filled = 5000
- 卖方：#1 成交 600/2000（remaining=1400），#2 未成交（remaining=2000），#3 未成交（remaining=2000）→ sell_filled = 600
- sell_aggressive = 0 → buy_excess = 5000 - 0 = 5000

撤单过程：

| 步骤 | 动作 | 释放量 | 累计释放 |
|------|------|--------|----------|
| 1 | 撤卖方 #2（完全未成交） | 2000 | 2000 |
| 2 | 撤卖方 #3（完全未成交） | 2000 | 4000 |
| 3 | 仍差 1000，撤卖方 #1（部分成交，remaining=1400） | 1400 | 5400 |

- 实际释放 5400 > excess 5000，多出 400
- 提交激进卖单：5000 股 @ bid_price_1
- 多出部分：400 股以 ask_price_1 重新挂被动卖单
- 更新：sell_aggressive += 5000

> **为什么先撤完全未成交的**：部分成交的子单已经在排队中拿到了一些好价格的成交，如果不撤它，剩余部分可能继续被动成交（这是最优情况）。完全未成交的子单没有任何排队优势，撤掉的"损失"最小。

**示例**：Block 买入总量 10000 股，卖出总量 10000 股，sub\_lots = 5（每个子单 2000 股）

| 事件                  | buy\_filled | sell\_filled | buy\_excess | sell\_excess | 动作                         |
| ------------------- | ---------- | ----------- | ---------- | ----------- | -------------------------- |
| 初始                  | 0          | 0           | 0          | 0           | 双边各 5 个子单被动挂单              |
| 买入 #1 全部成交 2000   | 2000       | 0           | 2000       | 0           | 撤 1 个未成交卖出子单，激进卖 2000        |
| 买入 #2 部分成交 1200   | 3200       | 0           | 1200       | 0           | 撤 1 个未成交卖出子单（释放 2000），激进卖 1200，多出 800 被动挂回 |
| 激进卖 2000 成交         | 3200       | 2000        | 0          | 0           | 差额为 0，无动作                  |
| 卖出 #3 被动成交 2000   | 3200       | 4000        | 0          | 1800        | 撤 1 个未成交买入子单（释放 2000），激进买 1800，多出 200 被动挂回 |
| …                   | —          | —           | —          | —           | 持续匹配直到双边都达到目标量             |

**关键优势**：

- **不关心哪个子单部分成交了多少** — 只看累计总量差额
- **子单只是撤单的操作句柄** — 需要撤多少量就撤多少个子单，干净利落
- **激进补单量精确等于差额** — 不多不少，买卖两边始终同步
- **自然处理任意部分成交模式** — 无论是子单全部成交、部分成交、还是多个子单同时部分成交，差额计算都正确

#### 3.5.4 激进补单与 Block 超时机制

Fill Matching 的被动→激进往复匹配通常在几秒~几十秒内自然收敛。作为安全网，设置整体超时限制：

**Block 超时参数**：`block_timeout`（默认 20 分钟），从 Block 启动（双边被动子单提交）开始计时。

**激进补单追价流程**（在超时之前）：

1. excess > 0 触发激进补单，提交后等待 `chase_wait_ticks` 个 tick
2. 等待期内成交 → 更新 filled，检查是否产生新的 excess
3. 等待期满仍未成交 → 撤单，以最新对方价格重新下单（追价）
4. 重复步骤 1-3，直到成交或触发超时

> **激进补单本身也可能部分成交**：这没关系。部分成交会更新 filled，下一次 on\_trade 回调时会重新计算 excess。系统自然收敛。

> **chase\_wait\_ticks 的 tick 计数基准**：基于主品种的 on\_tick 回调计数。因为 BacktestEngine 以主品种驱动策略循环，所有计时都在 on\_tick 中递增主品种的 tick 计数器。即使当前 Block 交易的是其他品种，追价等待仍然按主品种 tick 节奏计时。这确保了统一的时间基准。

> **订单被拒处理**：框架的 `buy()/sell()` 在非交易时间返回空字符串，订单也可能被 MatchingEngine 以 REJECTED 状态回调。处理方式：
> - `buy()/sell()` 返回 `""` → 视为下单失败，在下一个 on\_tick 中重试
> - `on_order()` 收到 `status=REJECTED` → 记录 ORDER\_REJECTED 日志，更新子单状态，在下一个 on\_tick 中以最新盘口价重新提交
> - 撤单请求发出后订单在撤单到达前已成交 → 无害，on\_trade 会正常更新 filled，excess 计算自动修正

#### 3.5.5 Block 超时处理

当 block\_timeout 到期时 Block 还未完成（buy\_filled < buy\_target 或 sell\_filled < sell\_target），执行以下分支：

**情境 A：零成交超时**（buy\_filled = 0 且 sell\_filled = 0）

- 双边被动单挂了 20 分钟无任何成交，市场流动性可能极差
- 操作：撤销所有未成交子单，完全放弃此 Block
- 状态 → TIMEOUT
- **无害**：没有任何成交发生，持仓和现金未受影响
- 记录 BLOCK\_TIMEOUT 日志

**情境 B：部分成交超时**（至少一边有成交）

- 正常的 Fill Matching 反复匹配未能在 20 分钟内收敛
- 操作：将双边所有剩余未成交量**强制转为激进单**
  - 买方未完成部分：撤掉剩余被动买单，以 ask\_price\_1 激进买入
  - 卖方未完成部分：撤掉剩余被动卖单，以 bid\_price\_1 激进卖出
- 激进单应在数秒内成交（因为是以对手价成交）
- 如激进单成功：状态 → DONE

**情境 C：激进单仍不能成交**（CRITICAL 事件）

这种情况极其罕见（以对手价挂单应该几乎肯定立即成交），如果发生说明市场出现了极端异常：

1. 记录 **CRITICAL** 级别日志，包含完整上下文（双边已成交量、未成交量、当前盘口快照）
2. 撤销所有剩余未成交的激进单
3. 已成交部分形成单边敞口，记录在 BlockTradeLog 中，等待后续交易周期自然修复
4. 状态 → CRITICAL
5. 发出告警（实盘时可连接外部告警系统）

> "单边敞口"并不可怕 — 加昂的一边已被卖出，便宜的一边买不进来，策略只是暂时持有更多现金。下一个信号周期会自然尝试用这笔现金买入最优标的。

#### 3.5.6 Block 交易状态机

```
BlockTrade 状态:
    PENDING    → 双边被动子单已提交，等待成交
    MATCHING   → 有成交发生，正在进行 excess 匹配和激进补单
    CHASING    → 激进补单提交后等待成交中
    DONE       → buy_filled ≥ buy_target 且 sell_filled ≥ sell_target
    TIMEOUT    → 超时且无任何成交，已撤销所有子单（无害放弃）
    CRITICAL   → 超时后激进单仍无法成交，存在单边敞口（极其罕见）
```

#### 3.5.7 执行流程图

```
信号触发 (Score > K, Dev > B, 不在冷却期)
    │
    ▼
计算 block_size = (当前现金 - min_cash_reserve) / cash_blocks
计算买入总量 / 卖出总量 → 拆分为 sub_lots 个子单
启动 block_timeout 计时器
    │
    ▼
┌─────────────────────────────────────────┐
│  同时提交双边各 sub_lots 个子单            │
│  Buy  #1..#n Si @ bid_price_1（被动挂单） │
│  Sell #1..#n H  @ ask_price_1（被动挂单） │
│  状态 → PENDING                          │
└──────────────┬──────────────────────────┘
               │
               ▼
     ┌───── 主循环：每个 on_tick / on_trade 检查 ─────┐
     │                                              │
     │  on_trade 回调：                             │
     │    更新 buy_filled / sell_filled              │
     │    计算 excess                              │
     │    excess > 0 → 撤被动子单，提交激进补单     │
     │                                              │
     │  on_tick 检查：                              │
     │    双边都完成？ → DONE                    │
     │    激进单超过 chase_wait_ticks 未成交？    │
     │      → 撤单重新以最新价格提交（追价）      │
     │    block_timeout 到期？                     │
     │      → 进入超时处理（3.5.5）              │
     │                                              │
     └──────────────────────────────────────────────┘

超时分支：
    ├─ 零成交 → 撤销全部 → TIMEOUT
    ├─ 有成交 → 强制激进 → DONE
    └─ 激进失败 → 撤销剩余 → CRITICAL
```

#### 3.5.8 Block 交易的串行原则

**核心规则：同一时刻只允许一个活跃的 Block 交易，必须执行完毕后才能启动下一个。**

Block 交易一旦启动（被动子单已提交），无论后续是否发现更好的机会，都必须跑完整个流程（被动 → 激进 → 追价 → 完成/超限）。

**不允许中途取消的原因**：

- 部分成交后已形成单边敞口，撤单会把敞口永久化
- 撤单本身也有延迟，可能撤单时又成交了一部分，状态更复杂
- “更好的机会”在几秒后可能就消失了，为了它取消正在执行的交易得不偿失

**对信号的处理**：

- Block 交易进行中时，信号计算照常运行（每个 tick 仍然计算 Score/Dev）
- 但不会启动新的 Block 交易 — 等同于冷却期内的行为：记录信号但不下单
- 当前 Block 完成后，如果新信号仍然有效，则在下一个 tick 触发新的 Block 交易（如不在冷却期内）

**典型时序**：

```
tick 100: 信号触发，启动 Block #1 交易（卖 A 买 B）
tick 101: Block #1 被动挂单中... 发现“卖 A 买 C”更优 → 记录但不行动
tick 102: Block #1 买方成交，卖方进入激进补单...
tick 104: Block #1 卖方成交，Block #1 完成 → 进入冷却期 1
tick 105: “卖 A 买 C”信号仍有效，但在冷却期内 → 等待
...
tick 300+: 冷却期结束，信号仍有效 → 启动 Block #2
```

#### 3.5.9 纯买入模式（无卖出的 Block 交易）

当满足以下条件之一时，Block 交易为纯买入模式（无卖出一边）：

- **空仓初始化**：策略启动后无持仓，需要建仓（择股算法见 4.4 节）
- **有剩余现金 Block**：cash\_blocks > 0 且信号指示某标的被低估，但当前无持仓标的需要卖出

纯买入模式没有"另一边"需要匹配，因此 Fill Matching 不适用。其执行逻辑独立于换仓模式，分为两个阶段：

##### 阶段一：未成交期（可换股）

- 以 bid\_price\_1（被动价格）提交 sub\_lots 个买入子单
- **只要还没有任何成交**，可以灵活调整：
  - 每个 on\_tick 重新执行择股算法（4.4.1 节）
  - 如果最优标的变了 → 撤掉全部挂单，以新标的重新下单
  - 如果 bid\_price\_1 变了 → 撤掉全部挂单，以新价格重新挂单（价格刷新）
  - 如果最优标的没变且价格没变 → 继续等待
- 此阶段无时间限制 — 因为没有任何成交发生，完全无害

##### 阶段二：部分成交后（计时器启动）

- 一旦收到第一笔成交回报（buy\_filled > 0），**立刻启动 block\_timeout 计时器**（20 分钟）
- 此后**不再允许换股**（已买入部分股份，换股会造成计划外持仓）
- 继续以 bid\_price\_1 被动挂单，等待剩余子单成交
  - 如果 bid\_price\_1 变动 → 撤旧单以新 bid\_price\_1 重新挂单（价格刷新，保持被动）
- **block\_timeout 到期且尚未全部成交**：
  - 撤掉所有剩余被动买单
  - 以 ask\_price\_1（激进价格）提交剩余量的买单
  - 激进买应在数秒内成交 → 状态 → DONE
  - 如激进买仍不成交 → CRITICAL 事件（同 3.5.5 情境 C 处理）

**状态机**：

```
纯买入 BlockTrade 状态:
    PENDING_BUY  → 被动买单已提交，等待成交（可换股、可刷新价格）
    FILLING      → 部分成交，计时器已启动，被动等待剩余成交（可刷新价格，不可换股）
    DONE         → buy_filled ≥ buy_target
    TIMEOUT      → 无（纯买入不存在零成交超时 — 阶段一无限等待，阶段二有 block_timeout）
    CRITICAL     → 超时后激进买单仍无法成交（极其罕见）
```

**与换仓模式的区别**：

- 没有 Fill Matching — 不存在"另一边"需要匹配
- 阶段一可以换股 — 换仓模式一旦启动不可取消
- 计时器从首笔成交开始 — 换仓模式从 Block 启动开始
- BlockTradeLog 中 sell\_symbol 为空，sell 相关字段为 0

> 纯买入主要出现在策略启动的前几个交易日（空仓建仓期），稳态运行后绝大多数 Block 交易是换仓模式。

### 3.6 T+1 限制

A 股 ETF 实行 T+1 交收制度：当日买入的份额不能当日卖出。

- 当日新买入份额标记为 today\_bought
- 可卖量 = 总持仓 - 冻结 - today\_bought
- 框架 MatchingEngine 已内置此机制，策略无需额外处理
- T+1 是冷却期之外的天然风控：即使冷却期全部结束，当日买入的份额仍不可卖出

### 3.7 Block 交易日志（Trade Log）

每个 Block 交易从创建到完成（或超限放弃），全程记录一份结构化日志。用于事后分析每笔交易是否合理、执行质量如何、策略逻辑是否有改进空间。

#### 3.7.1 日志数据结构

每个 Block 交易对应一个 `BlockTradeLog`：

```
BlockTradeLog:
    # ── 交易标识 ──
    block_id:         str         # 唯一编号，如 "20260310-001"
    trade_date:       date        # 交易日期
    block_seq:        int         # 当日第几个 Block（1, 2, 3...）

    # ── 决策快照（为什么交易） ──
    signal_time:      datetime    # 信号触发时刻（tick 时间戳）
    sell_symbol:      str         # 卖出标的
    buy_symbol:       str         # 买入标的
    trigger_score:    float       # 触发时的 Score(H, Si)
    trigger_dev:      float       # 触发时的 Dev
    trigger_max_dev:  float       # 触发时的 MaxDev
    trigger_ratio:    float       # 触发时的 Ratio_N
    mu:               float       # 当日的 μ
    candidate_list:   list        # 所有满足条件的候选列表 [(symbol, score, dev), ...]
    chosen_reason:    str         # 选择该候选的原因（如 "Dev 最大" 或 "近优候选 δ=0.08"）

    # ── 市场快照（下单时刻的盘口） ──
    buy_bid1:         float       # 买入标的下单时的 bid_price_1
    buy_ask1:         float       # 买入标的下单时的 ask_price_1
    sell_bid1:        float       # 卖出标的下单时的 bid_price_1
    sell_ask1:        float       # 卖出标的下单时的 ask_price_1

    # ── 执行参数 ──
    block_size:       float       # 本次 block_size（(现金 - min_cash_reserve) / cash_blocks）
    cash_blocks:      int         # 下单时剩余 cash_blocks
```

- **文件命名**：`block_trades_{date}.jsonl`，每个交易日一个文件

### 3.8 尾盘与收盘处理

A 股 ETF 14:57 进入收盘集合竞价阶段，限价单行为发生变化（不再即时撮合，统一收盘价成交）。策略需要在尾盘采取保守措施：

**交易截止时间**：`trading_cutoff`（默认 14:55:00），在此时间之后不启动新的 Block 交易。

**Block 时间安全检查**：启动新 Block 前检查"当前时间到 trading\_cutoff 的剩余分钟数"，若剩余时间 < block\_timeout 则不启动。这避免了 Block 刚启动就被截止时间强制超时、产生不必要的 CRITICAL 状态。

**尾盘规则**：

1. **14:55 后停止新 Block**：`tick.datetime.time() >= trading_cutoff` 时，即使信号满足也不启动新 Block
2. **活跃 Block 的处理**：如果 14:55 时有活跃的 Block 交易：
   - 立即触发超时处理（同 3.5.5 节），不等 block\_timeout 到期
   - 零成交 → 撤销全部 → TIMEOUT
   - 有成交 → 强制激进 → 争取在收盘前完成
   - 激进失败 → CRITICAL → 记录日志
3. **14:57 前必须清理完毕**：所有未成交订单在 14:56:50 前全部撤销，不留任何挂单进入集合竞价

> **设计意图**：策略的优势在盘中连续竞价阶段，集合竞价的价格形成机制不同，不应让策略订单参与集合竞价。2 分钟（14:55~14:57）足够完成强制激进和撤单清理。

## 4. 仓位管理（Position Management）

### 4.1 多标的分仓规则

| 参数                  | 说明        | 默认值 |
| ------------------- | --------- | --- |
| max\_positions      | 最大同时持仓标的数 | 3   |
| max\_single\_weight | 单标的最大仓位占比 | 50% |
| min\_cash\_reserve  | 最低现金保留金额（元），永久锁定  | 100  |

### 4.2 Block 分配逻辑

每次交易以一个 Block 为单位：

1. block\_size = (当前现金 - min\_cash\_reserve) / cash\_blocks（见 3.2.1 节）
2. 买入量 = round\_down(block\_size / buy\_price, 100)（对 100 股取整）
3. 单标的 block\_count 不超过 max\_block\_per\_symbol = floor(num\_blocks × max\_single\_weight)
4. 卖出量 = 持仓总量 / block\_count（见 3.2.2 节）
5. 买卖总量拆分为 sub\_lots 个子单（见 3.5.1 节）

当多个候选同时触发时：

- 按 Dev 降序排列，选择 Dev 最大的标的作为当前 Block 的买入目标
- 近优候选（Dev 在最佳的 (1-δ) 范围内）也有资格，但每次只执行一个 Block
- 多个候选需要分多个 Block 分别交易，受冷却期约束
- 单标的总 block\_count（含已有 + 新买）不超过 max\_block\_per\_symbol

### 4.3 换仓逻辑

换仓在每个 on\_tick 中按以下流程处理：

**第一步：信号收集**

1. 遍历所有持仓标的 H
2. 对每个 H，遍历所有非 H 标的 Si，计算 Score(H, Si)
3. 所有满足触发条件的 (H → Si) 对加入候选列表

**第二步：信号冲突安全检查**

1. 扫描候选列表，检测是否存在冲突模式（见 2.8 节）
2. 如检测到冲突 → 记录 NETTING_CONFLICT 日志（不影响执行）
3. 按 Dev 降序排列所有候选 (H → Si) 对

**第三步：执行状态检查**

1. 检查是否有活跃的 Block 交易进行中 → 如有，记录信号但不下单（串行原则 3.5.8 节）
2. 检查当日已交易 Block 数和冷却期状态
3. 如在冷却期内 → 记录信号但不下单，等待冷却结束
4. 无活跃 Block 且不在冷却期 → 进入步骤四

**第四步：Block 交易执行**

1. 选择最优卖出标的和买入标的（按 Dev 排序 + 择股原则 3.4 节）
2. 计算 block\_size = (当前现金 - min\_cash\_reserve) / cash\_blocks，确定买卖数量，拆分子单
3. 按 3.5 节双边被动挂单 + 子单配对策略执行
4. 成交后更新 block\_count、冷却期计数
5. 卖出资金即时可用于买入（A 股资金 T+0）

### 4.3.1 T+0 交易支持

部分 ETF 品种支持 T+0 交易（当日买入可当日卖出），策略通过**双重门控**机制支持此特性：

| 层级 | 参数 | 含义 |
| --- | --- | --- |
| 引擎层 | `enable_t0` | 品种"能不能" T+0 — 引擎是否允许当日买入的仓位当日卖出 |
| 策略层 | `enable_t0` | 策略"要不要" T+0 — 策略是否利用当日买入的仓位参与卖出信号 |

**只有两者都为 True 时，当日买入的仓位才可当日卖出。** 任一为 False，系统行为等同 T+1。

**实现机制**：

- 引擎层通过 `Position.today_bought` 追踪当日买入量，`Position.available = volume - frozen - today_bought`
- `enable_t0=True` 时，引擎不累加 `today_bought`，使得 `available` 自然包含当日买入量
- 策略代码中使用 `pos.available` 计算可卖量，无需额外分支

> **默认行为**：`enable_t0=False`（T+1 模式），与 A 股常规 ETF 交易规则一致。

### 4.4 空仓初始化与纯买入择股

策略启动时持仓为空，或有剩余 cash\_blocks 但无卖出信号时，信号模型（2.4 节）的 Score(H, Si) 无法计算（因为没有持仓 H）。此时使用替代的择股逻辑：

#### 4.4.1 空仓择股算法

**目标**：找到“如果持有它，不会立刻被信号要求卖出”的标的 — 即相对于标的池整体最被低估的品种。

**算法**：

1. 对标的池中每只标的 C，**假设持有 C**，扫描所有其他标的 Si，检查 Score(C, Si) 是否触发卖出条件（2.4 节四条全满足）
2. **安全标的**：如果不存在任何 Si 使得 Score(C, Si) 触发卖出条件 → C 是“安全”的（如果持有它，不会立刻被信号赶出去）
3. **不安全标的**：如果存在某个 Si 使得 Score(C, Si) 触发卖出 → C 是“贵”的，不应该买入
4. 在所有安全标的中，计算每只 C 对所有其他标的的 Dev 总和：
   `
   total_dev[C] = Σ Dev(C, Si)    对所有 Si ≠ C
   `
   total\_dev 越小（越负），说明 C 相对于整个标的池越被低估
5. 选择 total\_dev **最小**（最负）的安全标的作为买入目标

**异常处理**：如果没有任何安全标的（所有品种都“贵”到会被卖出信号赶走），这意味着信号参数可能有问题，或者市场处于极端状态：

- 记录 NO\_SAFE\_CANDIDATE 日志
- 不执行任何买入操作，保持全现金等待
- 下一个 tick 继续检查，直到出现安全标的

**示例**：标的池 5 只 ETF（A, B, C, D, E），全现金起步

| 标的 | 假设持有后是否触发卖出？ | total\_dev | 说明 |
|------|------------------------|-----------|------|
| A | 不触发 | -0.032 | 安全，中等低估 |
| B | 触发（Score(B,D) > K） | — | 不安全，排除 |
| C | 不触发 | -0.051 | 安全，**最被低估** |
| D | 不触发 | -0.028 | 安全，轻微低估 |
| E | 触发（Score(E,A) > K） | — | 不安全，排除 |

→ 选择 C 作为首个买入标的。

#### 4.4.2 空仓建仓流程

- 初始状态：cash\_blocks = num\_blocks（全部 Block 在现金中）
- 等待第一个 on\_day\_begin 完成日线数据加载和统计量计算
- 在第一个交易日的 tick 循环中，执行空仓择股算法找到最优标的
- 按纯买入模式（3.5.9 节）逐个 Block 建仓
  - 每个 Block 的 block\_size = (现金 - min\_cash\_reserve) / cash\_blocks，随着买入，cash\_blocks 递减
  - 受冷却期约束，不会一次性全部买入
  - 每次冷却期结束后重新执行择股算法，可能选择不同标的建仓（分散持仓）
- 建仓完成后（cash\_blocks 降到策略目标水平），转入正常换仓运行模式

## 6. 策略生命周期

```
on_init()
    ├── 加载所有品种的日线数据（前 N-1 天）
    ├── 预计算两两 Ratio 历史序列
    ├── 计算每对的 μ、Max、MaxDev
    └── 初始化状态：block_count 字典、冷却期计数器、持仓记录

on_day_begin(bar)                        # 每个交易日开盘前调用
    ├── 将昨日收盘价追加到 Ratio 历史（滚动窗口前移一天）
    ├── 重新计算 μ、Max、MaxDev（盘中不再变化）
    ├── 从 bar 获取当日 adj_factor（用于 tick→后复权转换）
    ├── 重置当日状态：冷却期计数归零、当日 Block 交易计数归零
    ├── 【跨日状态保留（不重置）】
    │   ├── block_count[symbol] — 跨日保持，反映实际 Block 分配
    │   ├── cash_blocks — 跨日保持（= num_blocks - Σ block_count）
    │   └── 持仓记录 — 由框架 Position 对象自动维护
    └── 【跨日一致性校验】
        ├── 比对 block_count 与实际持仓：Σ block_count 应 = 有持仓的标的数量合理范围
        ├── 如发现 CRITICAL/TIMEOUT 导致 block_count 与持仓不符 → 记录 INTEGRITY_WARN 日志
        └── v1.0 仅记录日志不自动修复，人工确认后手动调整

on_tick(tick)                            # 每个 tick 到来时调用
    ├── 过滤无效 tick（last_price <= 0 或在交易时间窗口外 → 跳过）
    ├── 收集所有品种最新价格
    │   ├── 主品种：tick.last_price
    │   └── 其他品种：get_latest_tick(symbol).last_price
    ├── 每个价格 × adj_factor → 后复权口径
    ├── 对每个持仓 H，计算 H 对所有非 H 标的的 Ratio_N → Dev → Score
    │
    ├── 【信号收集与安全检查】
    │   ├── 收集所有满足触发条件的 (H → Si) 对
    │   ├── 冲突检测（Netting Check）→ 如有冲突记录日志（见 2.8 节）
    │   └── 按 Dev 排序选择最优 (H, Si) 对
    │
    ├── 【执行状态检查】
    │   ├── 检查是否有活跃 Block 交易 → 如有，跳过下单（串行原则）
    │   ├── 检查当日已交易 Block 数和冷却期剩余时间
    │   └── 如在冷却期内 → 跳过下单，等待冷却结束
    │
    ├── 【Block 交易执行】
    │   ├── 选择最优卖出/买入标的（Dev 排序 + 择股原则）
    │   ├── 计算 block_size = (当前现金 - min_cash_reserve) / cash_blocks
    │   ├── 拆分为 sub_lots 个子单，双边被动挂单
    │   └── 进入 Fill Matching 流程（累计成交量匹配）
    │
    └── 【Fill Matching 监控】
        ├── on_trade 回调 → 更新 filled，计算 excess
        ├── excess > 0 → 撤被动子单，提交激进补单
        ├── 追价流程（chase_wait_ticks 间隔重新报价）
        ├── block_timeout 到期 → 超时处理（3.5.5 节）
        └── Block DONE 时更新 block_count、cash_blocks、冷却期计数

on_order(order)
    └── 记录订单状态变化（成交/撤单/拒单）

on_trade(trade)
    ├── 更新 BlockTrade 的 buy_filled / sell_filled
    ├── 计算 excess，触发激进补单（如需要）
    ├── 检查 Block 是否完成（双边均达标）
    │   └── Block DONE 时：更新 block_count[symbol]（买入 +1，卖出 -1），
    │       更新 cash_blocks，启动冷却期计时
    └── 更新持仓与资金状态

on_stop()
    └── 输出策略运行统计（总交易次数、换仓次数、冷却期触发次数等）
```

## 7. 与现有框架的适配

### 7.1 多品种 Tick 驱动

BacktestEngine 的 tick 模式实现：

- 主品种（symbols\[0]）驱动 on\_tick 回调
- 其他品种的 tick 同步推进到 ≤ 主品种时间戳
- 策略通过 get\_latest\_tick(symbol) 获取其他品种最新 tick

主品种充当"时钟"角色：即使当日对主品种无交易需求，它的 tick 仍驱动策略循环检查所有品种对。建议选日均成交额最大的品种做主品种，或由用户通过 primary\_symbol 参数指定。

### 7.2 日线数据访问

策略内部持有 ParquetBarFeed 实例：

- on\_init 时加载所有品种的日线数据
- on\_day\_begin 时引擎传入当日 BarData，策略追加到内部日线序列

### 7.3 撮合模式

回测使用 SMART\_TICK\_DELAY\_FILL 模式：

- 下单后不会在当前 tick 立即成交，而是延迟到下一个 tick
- 以下一个 tick 的盘口价格撮合，更真实地模拟实际交易延迟
- 限价单只有在盘口价格满足时才成交

## 8. 现有方案的问题与修改计划

### 8.1 窗口长度 N 的敏感性

**问题**：N=20（约一个月）是否对所有市场环境都适用？牛市波动大时可能需要更短的窗口来快速响应；震荡市可能需要更长窗口来过滤噪音。

**计划**：

- v1.0 用固定 N=20 实现
- 回测时做 N 的参数扫描（10、15、20、30、40）
- 未来考虑自适应 N：根据近期波动率自动调整窗口长度

### 8.2 Block 交易的串行性

**问题**：Block 交易从被动挂单到最终完成可能耗时较长（被动等待 + 激进补单 + 追价，正常在秒级~分钟级收敛，极端情况可达 block\_timeout 即 20 分钟）。期间可能出现更优的交易机会。

**设计决策**：

- **严格串行，不并发**：同一时刻只允许一个活跃 Block 交易（见 3.5.8 节）
- **Block 一旦启动，必须执行到底**：即使发现更好机会也不中途取消
- 待当前 Block 完成 + 冷却期结束后，如果更好的机会仍然存在，自然会在后续 tick 中触发
- 这种设计牺牲了少量时效性，但换取了状态管理的简洁和交易执行的确定性

### 8.3 换仓期间的风险敞口

**问题**：双边被动挂单后，一边先成交触发另一边撤单改为激进吃单，在这个过程中策略存在"单边持仓"的风险敞口。

**计划**：

- v1.0 接受此风险（ETF 流动性好，激进吃单后大概率即时成交）
- 初始被动挂单争取更优价格，一边成交后立即切换激进模式闭合头寸
- 追价策略确保最终成交，敞口暴露时间限制在秒级
- 记录每次换仓的双边成交时间差，回测后评估实际影响

### 8.4 佣金与滑点的累积效应

**问题**：每个 Block 交易产生双边佣金（5 个 Block × 双边 = 10 次佣金事件，而一次性全仓只有 2 次）。如果 Score 频繁在阈值附近波动触发小额交易，佣金会快速累积侵蚀收益。

**计划**：

- 冷却期是第一道防线，限制同日下单频率
- 最小偏离 LeastBias 是第二道防线，Dev 太小不值得操作
- 回测时统计总佣金占比，如果超过年化收益的 10%，需要调高 LeastBias 或冷却期时长
- 考虑设置最小交易金额阈值（如 ≥ 5000 元），过滤掉过小的碎片交易

### 8.5 adj_factor 的日内一致性

**问题**：信号模型（2.5 节）要求当日 tick 价格乘以 adj_factor 转换到后复权口径，与历史 Ratio 在同一口径下计算。但当前框架在 tick 模式下 `on_day_begin` 传入的是**前一个交易日的 BarData**，其 adj_factor 是昨日的值。如果 ETF 在当日发生了份额折算（拆分/合并），当日的 adj_factor 会与昨日不同，导致 Ratio_N 计算偏移。

**现状**：ETF 的 adj_factor 变化极为罕见（份额折算可能数年才发生一次），v1.0 暂时使用前一日的 adj_factor，实际影响几乎为零。

**未来计划**（必须实现）：

- 方案 A：在 BacktestEngine 的 tick 模式中增加传递当日 adj_factor 的机制（需修改引擎接口）
- 方案 B：在当日第一个 tick 到来时，从 daily bar 数据中查询当日的 adj_factor
- 方案 C：在 `on_day_begin` 阶段直接从 ParquetBarFeed 加载当日的 BarData 获取 adj_factor
- 无论采用哪种方案，需要在 adj_factor 发生变化的交易日做回归测试验证正确性

## 9. 相关想法

### 9.1 Score 平滑

Tick 价格波动频繁，单个异常 tick（如大单冲击或错误报价）可能导致 Score 瞬间跳变触发交易。两种应对方式：

- **EMA 平滑**：对 Score 做短期指数移动平均（如 α = 2/(30+1)，相当于 30 个 tick ≈ 90 秒窗口），用平滑后的 Score\_ema 替代原始 Score 做阈值判断。优点是稳定；缺点是引入延迟，可能错过快速行情。
- **依靠冷却期本身**：不做平滑，冷却期天然限制了连续触发的频率。即使单个尖峰触发了一个 Block 交易，损失也只是一个 Block（总资产的 1/num\_blocks）的短期波动。

建议 v1.0 先不做平滑，回测后根据"噪音触发率"决定是否引入。

### 9.2 品种相关性预筛选

当前标的池全员参与，但如果池中有两只 ETF 相关性极低（如沪深 300 和纳斯达克 100），它们的 Ratio 可能没有均值回归特性，Signal 是噪音。

**想法**：在 on\_init 阶段计算标的池内所有品种对的历史 Ratio 自相关系数或半衰期。只对均值回归特性显著的品种对计算 Score。这相当于一个静态的、数据驱动的品种对筛选，而非主观 watchlist。

### 9.3 日内避开高波动时段

开盘后 15 分钟（9:30-9:45）和收盘前 15 分钟（14:45-15:00）是 ETF 波动最大的时段，tick 价格噪音多。可以设定一个交易时间窗口（如 9:45-14:45），在窗口外不执行任何操作。

### 9.4 仓位再平衡

当前设计只在 Score 触发新信号时调整仓位。但随着时间推移，持仓品种的市值占比会因涨跌自然漂移。例如初始等权 33%/33%/33%，一段时间后可能变成 50%/25%/25%。

**想法**：每日收盘前（如 14:50）做一次仓位检查，如果某标的占比超过 max\_single\_weight 则主动减仓再平衡，不需要 Score 触发。

### 9.5 多时间维度 Score 融合

当前只使用一个 N 天窗口。可以考虑同时计算短期（N=5）、中期（N=20）、长期（N=60）三个 Score：

- 短期 Score 高 + 长期 Score 低 → 短期过热，可能很快回归，谨慎操作
- 短期 Score 高 + 长期 Score 高 → 趋势性偏离，果断操作
- 多维 Score 的加权组合可能比单一 N 更稳健

## 10. 方法评价

### 10.1 理论基础

**均值回归假设**：策略的核心假设是同类 ETF 的价格比率具有均值回归特性。这在统计上对高相关性品种（如同一指数的不同 ETF，或沪深 300 vs 中证 500）是成立的，因为套利力量会约束价差。但对低相关性品种，比率可能随机游走，此时策略退化为噪音交易。

**强项**：

- 利用相对价值（Ratio）而非绝对价格，天然对冲了市场系统性涨跌
- Score 归一化使不同品种对的信号可比
- 统一的日线+Tick 计算模型，逻辑简洁

**弱项**：

- 假设均值回归成立，但在趋势市（单边上涨/下跌）中，Ratio 可能持续偏离不回归
- 窗口长度 N 是固定的，对不同市场环境的适应性有限

### 10.2 执行层评价

**Block 分块交易的优劣**：

优点：

- 模型简单：每次交易一个 Block，无复杂的多阶段状态机
- 冷却期控制节奏，避免过度交易，参数少（只有 2 个冷却期时长）
- 双边激进挂单 + 撤单调价策略，兼顾成交确定性和价格优化
- Block 跟踪清晰：每个标的上有几个 Block 一目了然，卖出量计算简单

缺点：

- Block 数量固定，灵活性有限 — 不能对特别强的信号一次性大额建仓
- 冷却期可能错过快速行情中的最佳时机
- 双边订单的撤单调价涉及多次订单操作，实盘中需处理网络延迟和撤单失败等异常

**关于冷却期**：cooldown\_1 = 10 分钟、cooldown\_2 = 15 分钟是比较保守的设定。ETF tick 频率约 3 秒/跳，10 分钟 ≈ 200 个 tick。对于均值回归策略（本质上是逆势操作），保守一些更安全。两次冷却后放开限制，因为 T+1 本身就是天然的风控屏障。

### 10.3 容量与适用性

- **适用标的**：ETF（流动性好、T+1、无涨跌停板或极少触发）。不适用于个股（流动性差、涨跌停频繁）
- **资金容量**：单只 ETF 日均成交额通常在数亿到数十亿元。策略每次操作一个 Block（总资金的 1/num\_blocks），适用于 10 万 \~ 500 万元级别的资金量。超过此规模可能需要更多的 Block 分块
- **标的池规模**：21 只 ETF 是合理的池大小。品种太少（<5）信号机会少；品种太多（>50）计算量增加且低相关品种对会产生噪音信号

### 10.4 回测注意事项

- **前视偏差**：日线数据的 μ、Max 使用前 N-1 天，不包含当日，无前视偏差。但 adj\_factor 使用的是当日值，如果回测框架在当日开盘前就提供了当日的 adj\_factor，则无偏差。
- **存活偏差**：标的池中的 ETF 都是当前仍在交易的品种。对于已退市或合并的 ETF，如果不纳入标的池，会高估策略表现。
- **滑点估计**：回测中使用 ask\_price\_1/bid\_price\_1 作为成交价，已包含一跳滑点。但大单可能穿透多档盘口，实际滑点更大。Block 分块交易可以缓解此问题（每次只操作总资金的 1/num\_blocks）。
